## 프로젝트 실행 방법

1. 패키지 설치

```bash
pnpm install
# 또는
npm install
```

2. 개발 서버 실행

```bash
pnpm dev
# 또는
npm run dev
```

3. 접속

```
http://localhost:3000
```

# 게시판

## 사용한 기술 스택

- **Next.js(App Router) + React 18 + TypeScript**
  - 이유: 라우팅/서버렌더링 구조와 타입 안전성 확보
- **@tanstack/react-query**
  - 이유: 서버 상태 캐시/동기화 및 무한 스크롤 처리
- **@tanstack/react-table + @tanstack/react-virtual**
  - 이유: 컬럼 리사이즈/가시성 제어 + 대량 목록 성능 최적화
- **Tailwind CSS**
  - 이유: 디자인가이드가 없는 환경에서 UI 밀도 높은 화면에서 빠른 스타일 적용
- **Radix UI**
  - 이유: 접근성/상호작용 기본기 보장 (Dialog/Select/Popover 등)
- **lucide-react**
  - 이유: 아이콘 일관성 유지

## 주요 구현 기능 요약

### 1) 인증/세션

- `/auth/login`으로 JWT 발급 후 저장
- `useSyncExternalStore` + 하이드레이션 가드로 SSR/CSR 불일치 방지
- 토큰을 런타임 메모리 + localStorage에 동기화
  - 이유: 새로고침에도 로그인 유지

### 2) 게시글 CRUD

- 목록 / 상세 / 작성 / 수정 / 삭제 구현
- **작성/수정은 동일한 페이지 UX**
  - 이유: 라우트 단위로 상태가 안정적으로 유지되고, 공유/접근성이 좋아짐
- 내 글 전체 삭제 기능 제공
  - 이유: API 스펙 반영, 관리 작업 단축

### 3) 목록 테이블 UX

- 컬럼 리사이즈 및 컬럼 숨김/보임 설정
- 컨테이너 폭에 맞춰 자동 보정(공백/스크롤 최소화)
  - 이유: 화면마다 다른 폭에서도 일관된 밀도 유지
- 가상화(react-virtual)로 무한스크롤 렌더링 부담 완화
  - 이유: 스크롤 시 불필요 DOM 생성 최소화

### 4) 검색/정렬/필터

- 제목/본문 검색, 카테고리 필터, 정렬 기준/방향 제공
- 필터 상태를 URL 쿼리와 동기화
  - 이유: 뒤로가기 시 동일 결과 복원

### 5) 무한 스크롤

- **내 글(private)**: 커서 기반 API 무한 스크롤
- **전체 글(public)**: 고정 Mock 데이터를 클라이언트에서 단계적으로 노출
  - 이유: API 스펙에 맞춘 데이터 흐름 구성

### 6) 금칙어 처리

- 제목/본문/태그 각각 금칙어 검사 및 필드별 오류 표기
  - 이유: 사용자에게 정확한 수정 위치를 안내

### 7) UI 레이아웃

- GNB/툴바/필터 영역 고정(sticky) + 리스트 영역 스크롤 분리
- 게시판/상세/작성 페이지 컨테이너 폭 통일
  - 이유: 화면 전환 시 레이아웃 쉬프트 최소화

# 차트

## 사용한 기술 스택

- **ECharts + echarts-for-react**
  - 이유: 다양한 차트 유형(바/도넛/스택형/멀티라인)을 안정적으로 구현하고, 툴팁/범례/축 포맷을 유연하게 제어하기 위함
- **@tanstack/react-query**
  - 이유: Mock API 응답을 캐시하고 불필요한 재요청을 줄이기 위함

## 주요 구현 기능 요약

### 1) 바/도넛 차트

- `/mock/weekly-mood-trend` 데이터를 감정별 평균 비율로 요약하여 **바/도넛 차트** 구성
  - 이유: 주간 변동치를 분포 관점으로 재해석해 “비율 구성”을 한눈에 보여주기 위함
- `/mock/popular-snack-brands`를 **바/도넛 차트**로 동시 제공
  - 이유: 같은 데이터를 다른 관점(길이/비율)으로 비교 가능하게 하기 위함

### 2) 스택형 바/면적 차트

- `/mock/weekly-mood-trend`를 감정별 누적 비율로 **스택형 바/면적** 구현
- `/mock/weekly-workout-trend`를 운동 항목별 누적 비율로 **스택형 바/면적** 구현
- 활성 시리즈만 합산해 100% 기준으로 정규화
  - 이유: 숨김 시에도 축/비율이 일관되게 유지되도록 하기 위함

### 3) 멀티라인 차트

- `/mock/coffee-consumption` (커피 잔수 ↔ 버그/생산성)
- `/mock/snack-impact` (스낵 수 ↔ 회의불참/사기)
- 팀별 **두 개의 라인**(좌측/우측 축) + **실선/점선**, **원형/사각형 마커**로 구분
- 툴팁에서 동일 팀의 좌/우 지표를 함께 제공
  - 이유: 팀 단위 상관관계를 빠르게 파악할 수 있도록 하기 위함

### 4) 공통 UX

- 범례에서 **색상 변경/숨김 토글** 지원
- 축 가이드 문구를 카드 하단에 명시
  - 이유: 데이터 해석에 필요한 맥락을 차트 주변에서 즉시 확인 가능하게 하기 위함

# 공통

## 사용한 기술 스택 (공통)

- **Next.js(App Router) + React 18 + TypeScript**
  - 이유: 라우팅/SSR 기본기 확보와 타입 기반 안정성
- **Tailwind CSS**
  - 이유: 재사용 UI 블록을 빠르게 구성하고 레이아웃 일관성 유지
- **Radix UI**
  - 이유: 접근성 기본 동작을 보장하는 UI 베이스 제공

## 주요 구현 기능 요약 (공통)

- **공용 HTTP 클라이언트**
  - 이유: 요청/에러 처리 규칙을 중앙화해 예측 가능한 데이터 흐름 확보
- **토큰 스토리지 분리**
  - 이유: 인증 상태와 API 호출 의존성을 느슨하게 만들어 결합도 축소
- **공용 UI 컴포넌트 분리**
  - 이유: 화면별 반복 요소를 공통 레이어로 모아 응집도 향상

## 폴더 구조 (FSD)

```
src/
├── app/        # 라우팅/페이지 엔트리
├── widgets/    # 독립 UI 블록 (PostsBoard, GlobalNav)
├── features/   # 기능 단위 (post-form, post-table, auth 등)
├── entities/   # 도메인 모델 (post)
└── shared/     # 공용 컴포넌트/유틸
```

- 이유: 변경 범위를 국소화하고, 기능별 응집도를 높이기 위하여 FSD구조 채택
